InputStream / OutputStream Reader / Writer RandomAccessFile

I/O 类库中有一个广泛使用的抽象概念 – “流”, 其代表任何有能力产生数据的数据源对象或者是有能力接收数据的接收端对象, “流”屏蔽了 I/O 中的许多细节。

往简单了说InputStream / OutputStream Reader / Writer RandomAccessFile 这三个类就是 nio（新io） 之前 java io 的所有内容。

Java中的I/O类库非常旁杂，对于不同的输入读取方法和类型有专门对应的类，而最后的可用的类则需要编程者自己进行组合, 装饰。

Java有两套 I/O 体系 (继承树) , 一套是基于字节 (byte) 的 InputStream / OutputStream 类, 另一套是基于字符 (char) 的 Reader / Writer 类。Reader / Writer 类的出现事实上是基于国际化的考量 (对Unicode编码的支持)

个人的一点粗浅见解是，各式各样的 InputStream / OutputStream Reader / Writer 以及他们的导出类，可以用“水库”来比喻。他们的构造器接受的参数一般是数据源 (直接的数据源或者其他持有数据源的“流”)，而最基本的 read(), write() 方法则相当于阀门， I/O 类的本质是信息源的包装类，或者套娃包装类 (这种结构被称为装饰器模式(pattern))，最后的效果就是读写操作的规范化，接口的统一，提高了开发的效率以及安全性。

标准输入输出

在java中，System.in 和 System.out 的本质是两个静态的装饰器类，装饰的核心类其实就是 InputStream **和 **OutputStream 。因此, 标准输入输出是基于字节的。

关于输入输出基于什么: 者取决于输入/输出 中最底层的方法的工作方式, 在java中就是 read() 和 write()。

InputStream 的 read() 方法每调用一次产生一次读取一个字节的行为。 InputStream 所有的读取行为都通过花式调用 read() 来实现。所以 InputStream 是基于字节的，或者说， InputStream 是操作字节的流。

默认情况下 System.in 表示控制台 (console) 输入, System.out 则表示输出到控制台。配套的还有 System.err 表示错误信息的去向。

System.err 和 System.out 都是提供可直接调用的方法的完善的类 (例如System.out.println())，而 System.in 在使用时需要额外的装饰器 (只能作为参数提供信息)。

可以使用 System 类中的静态方法 setIn() setOut setErr() 来对三个流重新定向。需要注意的是，应该养成在不再需要重新定向时，将三个流的定向改回默认的习惯。所以在重新定向之前，要保存默认定向的引用，最后调用set方法将其还原到默认状态, 例如:

新IO (nio)

为了提高IO效率，java开发者们在 java.io 包的基础上开发了一套 java.nio(new io) 包, 其性能的提升主要源自于 “通道(channel)” 和 “缓冲器(buffer)” 的引入, 因为这两个结构的配合使得java io “更接近操作系统执行 io 的方式” (Thinking in Java, 18.10)。

ps: java.io 已经通过在源码中调用 java.nio 得到了优化

在新 io 中, 通道 (channel) 负责保管内存中的数据，缓冲器 (buffer) 则负责提供接口 (进行交互)。

通道是一个包含煤层的矿藏，而缓冲器则是派送到矿藏中的卡车。卡车满载煤炭而归，我们再从卡车上获取煤炭。

字节流的编码问题

为了使得程序语言国际化，一套编码往往需要支持世界上大部分主流语言，使用单个字节编码单个字符的ASC2字符集一共只有0x00-0xFF256个节点，这比起世界上的字符远远不够。庞大的字符数量需要足够多的节点来编码，采用多个字节来编码一个字符就可以提供足够多的节点. UTF-8的大部分字符就是双字节编码。

所以在读取字节流时，需要用户显式的调用 java.nio.charset.Charset包中提供的解码方法。而基于字符的流会自动使用平台默认的编码进行解码/编码(也可以设置)。

在ByteBuffer中存取基本数据类型

基本数据类型io，其实就是ByteBuffer这个底层缓冲器对char, int, short, long, float. double 这些类型的支持，所有的ByteBuffer对象, 都有 asCharBuffer() asShortBuffer() asLongBuffer() asFloatBuffer() asDoubleBuffer()这五个方法增加一层抽象，让ByteBuffer看起来是对应的基本数据类型的缓冲器，而正真的转型不曾发生，这种处理被称为**视图缓冲器(view buffer)**。因此，对ByteBuffer类型的对象调用一次asCharBuffer() asShortBuffer() asLongBuffer() asFloatBuffer() asDoubleBuffer()后，其类型依然是 ByteBuffer。

第二段代码似乎在暗示buff.asCharBuffer().put("数据")将会返回一个CharBuffer类型的对象。因此，下面这种做法看起来是被允许的,当你试图将cBuff作为fc.write()的参数，也确实会有一个参数类型错误的异常，这似乎说明buff被你转型成了CharBuffer。

但是，buff.asCharBuffer()并没有返回一个新的对象, buff和cbuff指向的是同一个缓冲器。

事实上，asCharBuffer()方法就像一个滤镜，它的作用只是使buff.asCharBuffer()这个整体表现出 CharBuffer的行为，而不会使buff本身产生任何变化。虽然后来看到这些文字的人可能不知道冒险小虎队是什么，但是视图缓冲器的运作方式确实就像冒险小虎队的解密卡。

通过ByteBuffer和Channel进行(文本)文件读写的基本流程

![image-20200715213010814](E:\Typora image\image-20200715213010814.png)

基本流程: 查找文件 -> 读入文件 -> 修改文件 -> 写出文件

1. 创建File对象，通过FilenameFilter等辅助类锁定目标文件，获取绝对路径。
2. 通过路径创建InputStream对象，通过InputStream对象获得Channel对象。
3. 为ByteBuffer对象分配空间(创建ByteBuffer对象)。
4. 将Channel对象的内容读取到ByteBuffer对象中(如果只是添加文件内容，创建新ByteBuffer)。**这一步记得调用`ByteBuffer`的`flip()`方法。**
5. 关闭输入流
6. 处理ByteBuffer(修改文件)，这一步涉及编码问题。
7. 通过路径创建OutputStream对象，通过OutputStream对象获得Channel对象。
8. 将修改后的ByteBuffer通过Channel对象写入文件
9. 关闭输出流

到这里, 小文本文件的读写就基本讲完了

内存映射文件

在一些内存较小的设备中，每次修改文件都把整个文件读取到内存中会非常奢侈。在文件的大小大于内存大小的极度情况下，甚至无法操作文件。内存映射文件就是为了处理这个问题而诞生的。

内存映射文件就像你电脑的窗口，当你查看一个数百页的文本文件时，你的电脑窗口只能显示一部分内容。但是你可以自由的查看，修改整个文件的任意某个部分。

有了内存映射文件，我们可以假定整个文件都放在内存中，而且可以把他当作非常大的数组来访问。

内存映射文件基于一套特殊的通道和缓冲器:

内存映射读写是nio的进化形态, 其性能在绝大部分情况下比起非映射读写有压倒性的优势。

  1. The user process sees the file data as memory, so there is no need to issue read() or write() system calls.
  2. As the user process touches the mapped memory space, page faults will be generated automatically to bring in the file data from disk. If the user modifies the mapped memory space, the affected page is automatically marked as dirty and will be subsequently flushed to disk to update the file.
  3. The virtual memory subsystem of the operating system will perform intelligent caching of the pages, automatically managing memory according to system load.
  4. The data is always page-aligned, and no buffer copying is ever needed.
  5. Very large files can be mapped without consuming large amounts of memory to copy the data.

the explanatory note of endsWith(Path p) :

Tests if this path ends with the given path.

If the given path has N elements, and no root component, and this path has N or more elements, then this path ends with the given path if the last N elements of each path, starting at the element farthest from the root, are equal.

If the given path has a root component then this path ends with the given path if the root component of this path ends with the root component of the given path, and the corresponding elements of both paths are equal. Whether or not the root component of this path ends with the root component of the given path is file system specific. If this path does not have a root component and the given path has a root component then this path does not end with the given path.

If the given path is associated with a different FileSystem to this path then false is returned.

the explanatory note of endsWith(String s):

Tests if this path ends with a Path, constructed by converting the given path string, in exactly the manner specified by the endsWith(Path) method. On UNIX for example, the path “foo/bar“ ends with “foo/bar“ and “bar“. It does not end with “r“ or “/bar“. Note that trailing separators are not taken into account, and so invoking this method on the Path“foo/bar“ with the String “bar/“ returns true.

There are two options for patterns: glob and regex. glob is simpler and is actually quite powerful so you’ll be able to solve many problems using glob. In matcher, the **/ at the beginning of the glob expression means “all subdirectories,” and it’s essential if you want to match more than just Paths ending in the base directory because it matches the full path up until your desired result. The single * is “anything,” then a dot, then the curly braces indicate a list of possibilities—we are looking for anything ending with either .tmp or .txt. You can find further details in the getPathMatcher() documentation. matcher2 just uses *.tmp, which would ordinarily not match anything, but adding the map() operation reduces the full path to just the name at the end. Notice in both cases that dir.tmp shows up in the output even though it’s a directory and not a file. To only find files, you must filter for them as in the last Files.walk()

How to use glob

  • The * character matches zero or more characters of a name component without crossing directory boundaries.

  • The ** characters matches zero or more characters crossing directory boundaries.

  • The ? character matches exactly one character of a name component.

  • The backslash character (\) is used to escape characters that would otherwise be interpreted as special characters. The expression \\ matches a single backslash and “{“ matches a left brace for example.

  • The [ ] characters are a bracket expression that match a single character of a name component out of a set of characters. For example, [abc] matches "a", "b", or "c". The hyphen (-) may be used to specify a range so [a-z] specifies a range that matches from "a" to "z" (inclusive). These forms can be mixed so [abce-g] matches "a", "b", "c", "e", "f" or "g". If the character after the [ is a ! then it is used for negation so [!a-c] matches any character except "a", "b", or "c".

    Within a bracket expression the *, ? and \ characters match themselves. The (-) character matches itself if it is the first character within the brackets, or the first character after the ! if negating.

  • The { } characters are a group of subpatterns, where the group matches if any subpattern in the group matches. The "," character is used to separate the subpatterns. Groups cannot be nested.

  • Leading period/dot characters in file name are treated as regular characters in match operations. For example, the "*" glob pattern matches file name ".login". The Files.isHidden(java.nio.file.Path) method may be used to test whether a file is considered hidden.

  • All other characters match themselves in an implementation dependent manner. This includes characters representing any name-separators.

  • The matching of root components is highly implementation-dependent and is not specified.

Files.readAllLines()

Files.write()

write()有3个重载版本来处理不同类型的写入。

  1. public static Path write(Path path, byte[] bytes, OpenOption... options)

    这个版本被用来写入字节流。

    The options parameter specifies how the file is created or opened. If no options are present then this method works as if the CREATE, TRUNCATE_EXISTING, and WRITE options are present. In other words, it opens the file for writing, creating the file if it doesn’t exist, or initially truncating an existing regular-file to a size of 0.

    在默认情况下，如果写入的文件不存在，其将被创建，如果已经存在，其将被覆盖。
